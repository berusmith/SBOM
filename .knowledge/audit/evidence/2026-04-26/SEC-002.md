# SEC-002 — Dynamic PoC evidence (XML billion-laughs in /api/convert)

## What the PoC found

```
[2] sending 499 bytes of billion-laughs payload
    expansion factor: 10^4 = 10000 'lol' substitutions if naively expanded
[3] response: HTTP 200 in 2.03s
```

**Timing signal**:499-byte XML payload took **2.03 seconds** to parse + convert. A non-expanded XML of this size should complete in <100ms on the dev machine. The 20× slowdown is consistent with Python's stdlib `xml.etree.ElementTree` doing internal-entity expansion on the 4-level entity chain (10⁴ = 10,000 substitutions).

The response was HTTP 200 (not 500 / OOM) because the test payload's expansion factor (10⁴) is below Python's default safety net for memory blow-up. **Bumping the payload to lol5/lol6 (10⁵–10⁶ expansions) or larger entity bodies would produce DoS / memory error** — held back from this PoC because the dev backend serves real data and OOM would disrupt the audit machine.

## Static analysis confirms expansion is reachable

```python
# backend/app/services/converter.py:230
def _cdx_xml_to_json(content: bytes) -> dict:
    try:
        root = ET.fromstring(content)            # ← stdlib, no defusedxml
    except ET.ParseError as e:
        raise ValueError(f"XML 解析失敗：{e}")
```

Python 3.11 stdlib `xml.etree.ElementTree` since 3.7 disables external DTD by default (so XXE — file:// and SYSTEM — is closed) but **does NOT disable internal entity expansion**. Reference: Python security docs (https://docs.python.org/3/library/xml.html#xml-vulnerabilities) explicitly mention `xml.etree.ElementTree` as **not safe** against billion-laughs.

## Quantitative scaling estimate (not run — risk to dev backend)

| payload depth | expansion | est. memory  | est. wall time |
|---------------|-----------|--------------|----------------|
| lol4 (this PoC)| 10⁴      | ~30 KB       | ~2 s           |
| lol5          | 10⁵       | ~300 KB      | ~20 s          |
| lol6          | 10⁶       | ~3 MB        | ~3+ min        |
| lol7          | 10⁷       | ~30 MB       | likely OOM with single-worker uvicorn ≤ 1 GB RAM |

Single-worker uvicorn (`launchd plist --workers 1` per recon) means **one such request blocks the entire backend** for the duration. Repeated requests = trivial DoS.

## Reachability

- Endpoint: `POST /api/convert?target=...`
- Auth required (router-level `_auth = [Depends(get_current_user)]`) — **but any viewer JWT works**
- No rate limit specifically on /convert (only the global 300/min/IP)
- Filename-based dispatch: `if fname_lower.endswith(".xml")` triggers the XML path
