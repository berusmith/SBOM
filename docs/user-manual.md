# 顧問操作手冊

本手冊說明如何以 SBOM 管理平台完成一個完整的顧問案作業流程，從建立客戶到產出合規報告。

---

## 快速導覽

```
Step 1  建立客戶組織
Step 2  建立產品
Step 3  建立版本 & 上傳 SBOM
Step 4  掃描 CVE + 豐富化情資
Step 5  VEX 標注（排除假陽性）
Step 5b 漏洞抑制（風險接受）
Step 6  合規評估
Step 7  產出報告/證據包
Step 8  (視需要) 建立 CRA 事件
```

---

## Step 1 — 建立客戶組織

1. 左側選單 → **Organizations**
2. 點選右上角「新增組織」
3. 填入客戶公司名稱，授權狀態選 `trial`（正式收費後改為 `active`）
4. 按確認儲存

> 所有資料以組織為隔離單位，不同客戶的資料不會互相看見。

---

## Step 2 — 建立產品

1. 點進剛建立的組織
2. 點選「新增產品」
3. 填入產品名稱（對應客戶的設備型號，如：工業閘道器 GA-200）
4. 選填說明欄位

---

## Step 3 — 建立版本 & 上傳 SBOM

1. 點進產品，點選「新增版本」
2. 填入版本號（如：v2.1.0）
3. 版本建立後，進入版本詳情頁
4. 點選「上傳 SBOM」，選擇 `.json` 檔案
   - 支援格式：**CycloneDX JSON**、**SPDX JSON**
5. 上傳成功後系統自動解析元件清單，並顯示與前一版本的**差異摘要**（新增/移除的元件與漏洞數量）

**SBOM 格式要求：**
- CycloneDX：`bomFormat: "CycloneDX"`，`components[]` 每筆含 `name`、`version`
- SPDX：`spdxVersion: "SPDX-2.x"`，`packages[]` 每筆含 `name`、`versionInfo`
- 有 PURL（`purl` 欄位）的元件掃描準確度更高
- CycloneDX 若含 `dependencies[]` 區塊，可在「依賴關係圖」頁籤視覺化呈現

---

## Step 4 — 掃描 CVE + 豐富化情資

上傳 SBOM 後，在版本頁面依序執行：

| 按鈕 | 動作 | 說明 |
|------|------|------|
| 重新掃描 CVE | OSV.dev API 查詢 | 每個元件的 PURL 對應 CVE，結果秒到 |
| 更新 EPSS | FIRST.org API | 補充利用可能性分數（0–100%），評估優先處理順序 |
| 更新 NVD | NVD API 2.0 | 補充 CVE 描述、CWE、CVSS v3/v4、參考連結 |

> **效率提示：** 未設定 NVD API Key 時，NVD 豐富化每筆約需 7 秒。申請免費 API Key 後填入 `.env`，速度提升 10 倍。

掃描後即可在漏洞清單看到：
- 嚴重度（Critical / High / Medium / Low）
- CVSS v3/v4 分數
- EPSS 分數（數字越高 = 越常被利用）
- KEV 標記（CISA 已知被利用漏洞，必須優先處理）
- **SLA 欄位**（剩餘天數或逾期天數，見下方說明）

### SLA 追蹤

系統依嚴重度自動計算修補期限（從 `scanned_at` 起算）：

| 嚴重度 | SLA 天數 |
|--------|---------|
| Critical | 7 天 |
| High | 30 天 |
| Medium | 90 天 |
| Low | 180 天 |

- **紅色「逾 N 天」** — 已超過 SLA，需立即處理
- **橘色「剩 N 天」** — 剩餘不足 25%，即將到期
- 灰色數字 — 正常範圍內
- 已 `fixed` / `not_affected` 或已抑制的漏洞不顯示 SLA

---

## Step 5 — VEX 標注（排除假陽性）

SBOM 掃描常出現大量假陽性（元件雖有漏洞，但產品實際未使用該功能）。VEX 標注是顧問的核心加值服務。

**單筆更新：**
1. 點選漏洞列旁的「編輯」
2. 選擇狀態：
   - `open` — 尚未評估（預設）
   - `in_triage` — 評估中
   - `not_affected` — 不受影響（須選擇理由）
   - `affected` — 確認受影響（須選擇回應方式）
   - `fixed` — 已修補
3. 填入說明備註，按儲存

**批次更新（建議用於同一元件的多個 CVE）：**
1. 勾選多筆漏洞
2. 底部浮動列選擇狀態
3. 點「套用」一次更新所有勾選項目

**VEX not_affected 常用理由：**
- `code_not_present` — 程式碼未包含（常用於間接相依）
- `code_not_reachable` — 程式碼存在但執行路徑不可達
- `requires_configuration` — 需特定設定才能觸發（預設設定安全）

所有狀態變更自動記錄於 VEX 歷程，點選 CVE 編號可展開查看完整時間軸。

---

## Step 5b — 漏洞抑制（風險接受）

對於暫時無法修補、但已通過風險評估的漏洞，可使用**抑制**功能（區別於 VEX 標注）。

**使用時機：**
- 漏洞已評估，但廠商尚未提供修補程式
- 已與客戶確認風險可接受，等待下一季修補計畫
- 內部環境防護已到位，短期暫不修補

**操作方式：**
1. 點選漏洞列旁的「抑制」按鈕
2. 填入抑制原因（自由文字，供稽核使用）
3. 設定有效期限（選填）：到期後系統自動恢復計算
4. 按「確認抑制」

**抑制的影響：**
- 嚴重度統計（頁面頂部色塊）不計入抑制漏洞
- SLA 逾期統計不計入
- 發布品質閘門（Policy Gate）的 Critical 檢查不計入
- 漏洞仍保留在資料庫，切換「顯示已抑制」可查閱

**解除抑制：**
點已抑制漏洞的「已抑制」橘色按鈕 → 選「解除抑制」即可恢復計算。

## Step 6 — 合規評估

### 發布品質閘門（Policy Gate）

版本頁面頂部的「發布品質閘門」顯示 5 項檢查結果：

| 檢查項目 | 通過條件 |
|---------|---------|
| 無 Critical 未修補 | Critical 漏洞全部 fixed / not_affected（或已抑制） |
| 未分類率 < 20% | open 漏洞佔比低於 20% |
| 所有 KEV 已處置 | CISA KEV 漏洞無 open 狀態 |
| SBOM 品質 ≥ 70% | NTIA 7 項最低要求得分 ≥ 70% |
| 無 Policy Block 違規 | 無 block 等級的 Policy 規則被觸發 |

全部通過顯示綠色 **PASS**；任一未通過顯示紅色 **FAIL**。  
建議在產品發布前確認 PASS 狀態再鎖定版本。

### IEC 62443-4-1（開發流程）
版本頁面 → IEC 62443-4-1  
評估 11 項要求（SM-9、DM-1~5、SUM-1~5），依實際 DB 資料自動計算符合度。

### IEC 62443-4-2（元件安全）
IEC 62443-4-2  
評估 4 大類（CR-1 到 CR-4）12 項元件技術要求。

### IEC 62443-3-3（系統安全）
IEC 62443-3-3  
評估 7 大 Foundational Requirement（FR）下的 System Requirement。

### CRA 合規摘要
合規頁面顯示：
- Critical/High 未修補數量（Article 13：安全更新要求）
- VEX 覆蓋率（有明確狀態的漏洞比例）
- SBOM 完整性（上傳後有無被竄改）

## Step 7 — 產出報告與證據包

| 類型 | 按鈕 | 說明 |
|------|------|------|
| 主報告 | 下載 PDF | 含 Logo、客戶名、嚴重度分布、漏洞明細 |
| 漏洞清單 | 匯出 CSV | 可交給客戶或匯入其他系統 |
| CSAF VEX | 匯出 CSAF VEX | CSAF 2.0 格式，符合 EU CRA 要求的機器可讀格式 |
| 合規證據包 | 下載證據包 ZIP | 包含：PDF + CSAF + SBOM 原始檔 + manifest.json |
| IEC 報告 | 各別下載 | 4-1 / 4-2 / 3-3 三份獨立 PDF |
| CycloneDX XML | CycloneDX XML | 標準格式 SBOM 匯出 |
| SPDX JSON | SPDX JSON | 標準格式 SBOM 匯出 |

**版本鎖定：** 客戶確認後，點選「鎖定版本」防止資料被修改。可透過完整性驗證確認 SBOM 未被竄改。

## Step 8 — CRA 事件管理（Article 14）

當發現產品存在**已被積極利用**（KEV 標記或確認受影響）的漏洞時，依 EU CRA Article 14 需在規定時限內通報 ENISA。

**建立事件：**
1. 左側選單 → CRA Incidents → 新增事件
2. 填入標題、說明、觸發 CVE 編號（可複數）
3. 選擇觸發來源（`manual` / `kev` / `osv`）

**SLA 時鐘操作：**

| 步驟 | 操作 | 截止時限 |
|------|------|---------|
| 1 | 確認知悉（start-clock） | T+0 |
| 2 | 提交 Early Warning | T+24 小時 |
| 3 | 提交 Vulnerability Notification | T+72 小時 |
| 4 | 修補完成後 | 提交 Final Report（修補後 14 天內） |

> **重要：** `start-clock` 按鈕代表法律意義上的「知悉時間點」。確認要通報前再按，不可撤回。

事件關閉後，填入 ENISA 回傳的參考編號以完成稽核紀錄。

## 常見作業情境

### 情境 A：新版本上市前審查
1. 建立版本 → 上傳客戶提供的 SBOM
2. 掃描 + 豐富化
3. 進行 VEX 標注，排除假陽性
4. 確認 Policy Gate 全部通過
5. 產出 IEC 62443 報告 + 合規證據包
6. 鎖定版本

### 情境 B：CVE 緊急事件回應（如：Log4Shell）
1. CRA Incidents → 建立事件，填入 CVE-2021-44228
2. 全域搜尋：搜尋欄輸入 `log4j`，找出所有受影響產品版本
3. 各版本進行 VEX 評估：哪些版本真正受影響？
4. 啟動 SLA 時鐘，依時限提交 ENISA 報告
5. 修補後更新 VEX 狀態為 `fixed`，下載最新報告

### 情境 C：跨客戶週報
1. 左側選單 → Risk Overview
2. 查看各組織未修補 Critical/High 數量排行
3. Dashboard → 高風險元件表格，點選元件名稱直接搜尋
4. Dashboard → SLA 逾期計數（紅色顯示），優先處理超時案件
5. 點進排名最高的組織優先處理

### 情境 D：依賴關係審查
1. 進入版本詳情頁 → 點選「依賴關係圖」頁籤
2. 紅色節點 = 有 Critical/High 未修補漏洞的元件
3. 藍色節點 = 根節點（直接相依）
4. 點選節點查看元件名稱與版本，評估間接相依風險深度

## 品牌化設定（報告 Logo）

設定 → 品牌設定  
- 上傳公司 Logo（建議 PNG，背景透明，最大 2MB）
- 填入公司名稱和報告頁尾文字
- 主題色（HEX 色碼，影響 PDF 報告頁首顏色）

所有後續產出的 PDF 報告自動套用設定。
