# 顧問操作手冊

本手冊說明如何以 SBOM 管理平台完成一個完整的顧問案作業流程，從建立客戶到產出合規報告。

---

## 快速導覽

```
Step 1  建立客戶組織
Step 2  建立產品
Step 3  建立版本 & 上傳 SBOM
Step 4  掃描 CVE + 豐富化情資
Step 5  VEX 標注（排除假陽性）
Step 6  合規評估
Step 7  產出報告/證據包
Step 8  (視需要) 建立 CRA 事件
```

---

## Step 1 — 建立客戶組織

1. 左側選單 → **Organizations**
2. 點選右上角「新增組織」
3. 填入客戶公司名稱，授權狀態選 `trial`（正式收費後改為 `active`）
4. 按確認儲存

> 所有資料以組織為隔離單位，不同客戶的資料不會互相看見。

---

## Step 2 — 建立產品

1. 點進剛建立的組織
2. 點選「新增產品」
3. 填入產品名稱（對應客戶的設備型號，如：工業閘道器 GA-200）
4. 選填說明欄位

---

## Step 3 — 建立版本 & 上傳 SBOM

1. 點進產品，點選「新增版本」
2. 填入版本號（如：v2.1.0）
3. 版本建立後，進入版本詳情頁
4. 點選「上傳 SBOM」，選擇 `.json` 檔案
   - 支援格式：**CycloneDX JSON**、**SPDX JSON**
5. 上傳成功後系統自動解析元件清單

**SBOM 格式要求：**
- CycloneDX：`bomFormat: "CycloneDX"`，`components[]` 每筆含 `name`、`version`
- SPDX：`spdxVersion: "SPDX-2.x"`，`packages[]` 每筆含 `name`、`versionInfo`
- 有 PURL（`purl` 欄位）的元件掃描準確度更高

---

## Step 4 — 掃描 CVE + 豐富化情資

上傳 SBOM 後，在版本頁面依序執行：

| 按鈕 | 動作 | 說明 |
|------|------|------|
| 開始掃描 | OSV.dev API 查詢 | 每個元件的 PURL 對應 CVE，結果秒到 |
| 豐富化 EPSS | FIRST.org API | 補充利用可能性分數（0–100%），評估優先處理順序 |
| 豐富化 NVD | NVD API 2.0 | 補充 CVE 描述、CWE、CVSS v3/v4、參考連結 |

> **效率提示：** 未設定 NVD API Key 時，NVD 豐富化每筆約需 7 秒。申請免費 API Key 後填入 `.env`，速度提升 10 倍。

掃描後即可在漏洞清單看到：
- 嚴重度（Critical / High / Medium / Low）
- CVSS 分數
- EPSS 分數（數字越高 = 越常被利用）
- KEV 標記（CISA 已知被利用漏洞，必須優先處理）

---

## Step 5 — VEX 標注（排除假陽性）

SBOM 掃描常出現大量假陽性（元件雖有漏洞，但產品實際未使用該功能）。VEX 標注是顧問的核心加值服務。

**單筆更新：**
1. 點選漏洞列旁的「編輯」
2. 選擇狀態：
   - `open` — 尚未評估（預設）
   - `in_triage` — 評估中
   - `not_affected` — 不受影響（須選擇理由）
   - `affected` — 確認受影響（須選擇回應方式）
   - `fixed` — 已修補
3. 填入說明備註，按儲存

**批次更新（建議用於同一元件的多個 CVE）：**
1. 勾選多筆漏洞
2. 點選「批次更新」
3. 選擇狀態和說明，一次套用

**VEX not_affected 常用理由：**
- `code_not_present` — 程式碼未包含（常用於間接相依）
- `code_not_reachable` — 程式碼存在但執行路徑不可達
- `requires_configuration` — 需特定設定才能觸發（預設設定安全）

所有狀態變更自動記錄於 VEX 歷程，可供稽核查詢。

---

## Step 6 — 合規評估

### IEC 62443-4-1（開發流程）
版本頁面 → 報告 → IEC 62443-4-1  
評估 11 項要求（SM-9、DM-1~5、SUM-1~5），依實際 DB 資料自動計算符合度。

### IEC 62443-4-2（元件安全）
報告 → IEC 62443-4-2  
評估 4 大類（CR-1 到 CR-4）12 項元件技術要求。

### IEC 62443-3-3（系統安全）
報告 → IEC 62443-3-3  
評估 7 大 Foundational Requirement（FR）下的 System Requirement。

### CRA 合規摘要
合規頁面顯示：
- Critical/High 未修補數量（Article 13：安全更新要求）
- VEX 覆蓋率（有明確狀態的漏洞比例）
- SBOM 完整性（上傳後有無被竄改）

---

## Step 7 — 產出報告與證據包

| 類型 | 按鈕 | 說明 |
|------|------|------|
| 主報告 | 下載 PDF | 含 Logo、客戶名、嚴重度分布、漏洞明細 |
| 漏洞清單 | 匯出 CSV | 可交給客戶或匯入其他系統 |
| CSAF VEX | 下載 CSAF | CSAF 2.0 格式，符合 EU CRA 要求的機器可讀格式 |
| 合規證據包 | 下載 ZIP | 包含：PDF + CSAF + SBOM 原始檔 + 清單 manifest.json |
| IEC 報告 | 各別下載 | 4-1 / 4-2 / 3-3 三份獨立 PDF |

**版本鎖定：** 客戶確認後，點選「鎖定版本」防止資料被修改。可透過完整性驗證確認 SBOM 未被竄改。

---

## Step 8 — CRA 事件管理（Article 14）

當發現產品存在**已被積極利用**（KEV 標記或確認受影響）的漏洞時，依 EU CRA Article 14 需在規定時限內通報 ENISA。

**建立事件：**
1. 左側選單 → CRA Incidents → 新增事件
2. 填入標題、說明、觸發 CVE 編號（可複數）
3. 選擇觸發來源（`manual` / `kev` / `osv`）

**SLA 時鐘操作：**

| 步驟 | 操作 | 截止時限 |
|------|------|---------|
| 1 | 確認知悉（start-clock） | T+0 |
| 2 | 提交 Early Warning | T+24 小時 |
| 3 | 提交 Vulnerability Notification | T+72 小時 |
| 4 | 修補完成後 | 提交 Final Report（修補後 14 天內） |

> **重要：** `start-clock` 按鈕代表法律意義上的「知悉時間點」。確認要通報前再按，不可撤回。

事件關閉後，填入 ENISA 回傳的參考編號以完成稽核紀錄。

## 常見作業情境

### 情境 A：新版本上市前審查
1. 建立版本 → 上傳客戶提供的 SBOM
2. 掃描 + 豐富化
3. 進行 VEX 標注，排除假陽性
4. 產出 IEC 62443 報告 + 合規證據包
5. 鎖定版本

### 情境 B：CVE 緊急事件回應（如：Log4Shell）
1. CRA Incidents → 建立事件，填入 CVE-2021-44228
2. 全域搜尋：搜尋欄輸入 `log4j`，找出所有受影響產品版本
3. 各版本進行 VEX 評估：哪些版本真正受影響？
4. 啟動 SLA 時鐘，依時限提交 ENISA 報告
5. 修補後更新 VEX 狀態為 `fixed`，下載最新報告

### 情境 C：跨客戶週報
1. 左側選單 → Risk Overview
2. 查看各組織未修補 Critical/High 數量排行
3. 點進排名最高的組織優先處理

---

## 品牌化設定（報告 Logo）

設定 → 品牌設定  
- 上傳公司 Logo（建議 PNG，背景透明，最大 2MB）
- 填入公司名稱和報告頁尾文字
- 主題色（HEX 色碼，影響 PDF 報告頁首顏色）

所有後續產出的 PDF 報告自動套用設定。
